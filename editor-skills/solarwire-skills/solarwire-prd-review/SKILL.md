# SolarWire PRD 评审 (solarwire-prd-review)

## Overview

对已完成的 SolarWire PRD 文档进行结构化评审，确保质量达标。

**重要：** 本技能依赖 `solarwire: solarwire-syntax` 进行语法验证，依赖 `solarwire: solarwire-prd` 的模板标准。

## When to Use

**症状和使用场景：**
- PRD 生成完成后需要质量检查
- PRD 修改后需要重新评审
- 交付开发前需要最终确认
- 团队协作时需要交叉评审

**反模式（不应使用本技能）：**
- PRD 还未生成 → 先使用 `solarwire: solarwire-prd`
- 仅需语法验证 → 使用 `solarwire: solarwire-syntax`

## Review Dimensions

### 1. 完整性检查

| 检查项 | 要求 | 检查方法 |
|--------|------|---------|
| 产品背景 | 必须说明业务背景 | 检查 1.1 章节 |
| 产品目标 | 必须有可衡量的目标 | 检查 1.2 章节 |
| 目标用户 | 必须明确用户群体 | 检查 1.3 章节 |
| 功能模块 | 必须有功能模块表 | 检查 2.1 章节 |
| 信息架构 | 必须说明导航结构 | 检查 2.2 章节 |
| 页面路径 | 每个页面必须有路径 | 检查 3.x.1 章节 |
| 页面功能 | 每个页面必须有功能说明 | 检查 3.x.1 章节 |
| 字段说明 | 每个页面必须有字段表 | 检查 3.x.3 章节 |
| 交互逻辑 | 每个页面必须有交互说明 | 检查 3.x.3 章节 |
| 异常处理 | 关键流程必须有异常处理 | 检查 3.x.3 章节 |

### 2. 语法验证

| 检查项 | 要求 | 检查方法 |
|--------|------|---------|
| SolarWire 代码块 | 每个 ` ```solarwire ` 代码块必须语法正确 | 调用 validate.js 验证 |
| 元素定位 | 每个元素必须有 `@()` 定位 | 代码审查 |
| 必填属性 | 宽高元素必须有 `w=` `h=` | 代码审查 |
| 颜色格式 | 必须使用 `#RRGGBB` 格式 | 代码审查 |
| 表格结构 | 必须先 `##` 后 `#` | 代码审查 |

### 3. 可测试性检查

| 检查项 | 要求 | 检查方法 |
|--------|------|---------|
| 字段类型明确 | 类型必须是标准类型 | 检查字段表 |
| 校验规则明确 | 必须有具体的校验条件 | 检查校验规则列 |
| 交互可验证 | 操作→响应对应关系明确 | 检查交互逻辑 |
| 异常可验证 | 异常场景和处理可测试 | 检查异常处理 |

### 4. 一致性检查

| 检查项 | 要求 | 检查方法 |
|--------|------|---------|
| 页面跳转闭环 | 所有入口都有对应出口 | 检查页面入口/出口 |
| 字段名称一致 | 同一字段在各页面名称一致 | 跨页面对比 |
| 交互行为一致 | 相同操作的响应行为一致 | 跨页面对比 |
| 术语一致 | 术语使用前后一致 | 全文检查 |

## Workflow

### Step 1: 读取 PRD
- 加载完整的 PRD 文档
- 识别所有章节结构

### Step 2: 完整性检查
- 逐项检查完整性检查表中的 10 个检查项
- 记录缺失或不完整的项目

### Step 3: 语法验证
- 提取所有 SolarWire 代码块
- 逐个代码块运行 `solarwire: solarwire-syntax` 的 validate.js 验证
- 记录语法错误位置和描述

### Step 4: 可测试性检查
- 逐项检查可测试性检查表中的 4 个检查项
- 记录无法编写测试用例的项目

### Step 5: 一致性检查
- 逐项检查一致性检查表中的 4 个检查项
- 记录矛盾或不一致的地方

### Step 6: 输出评审报告

按以下格式输出评审结果：

```markdown
# PRD 评审报告

## 评审概要
- **评审时间：** YYYY-MM-DD
- **评审文档：** PRD 文件路径
- **评审维度：** 完整性、语法、可测试性、一致性

## 评审结果

### 1. 完整性检查
| 检查项 | 结果 | 问题描述 | 建议修复 |
|--------|------|---------|---------|

### 2. 语法验证
| 检查项 | 结果 | 问题描述 | 建议修复 |
|--------|------|---------|---------|

### 3. 可测试性检查
| 检查项 | 结果 | 问题描述 | 建议修复 |
|--------|------|---------|---------|

### 4. 一致性检查
| 检查项 | 结果 | 问题描述 | 建议修复 |
|--------|------|---------|---------|

## 评审总结
- **通过项：** N/总项数
- **问题项：** N/总项数
- **评审评分：** XX/100

## 评审结论
- ✅ 通过 / ⚠️ 有条件通过 / ❌ 不通过
- **后续行动：** 需要修复的问题清单
```

## Scoring

| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 完整性 | 40% | 10 个检查项，每项 4 分 |
| 语法 | 30% | 全部通过 30 分，每错 1 处扣 5 分 |
| 可测试性 | 15% | 4 个检查项，每项 3.75 分 |
| 一致性 | 15% | 4 个检查项，每项 3.75 分 |

**评审等级：**
- ≥ 90 分：✅ 通过
- 70-89 分：⚠️ 有条件通过（需修复关键问题）
- < 70 分：❌ 不通过（需重大修改后重新评审）

## Red Flags

| Thought | Reality |
|---------|---------|
| "PRD 生成时已经检查过了" | 生成和评审是两个独立环节，必须单独评审 |
| "语法验证可以跳过，其他技能已经验证了" | 评审必须重新验证，不依赖其他环节的结果 |
| "这个页面很简单，不需要详细检查" | 每个页面都必须按标准完整检查 |
| "用户已经确认过需求了" | 用户确认不等于结构完整性和语法正确 |

## Real-World Impact

通过严格的评审流程：
- 减少 70%+ 的需求遗漏问题
- 确保 100% 的 SolarWire 代码可正确渲染
- 避免开发阶段的需求澄清返工
- 为测试用例生成提供可靠基础
