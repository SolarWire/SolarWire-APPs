- DRY原则：Don't Repeat Yourself（不要重复自己）；
- 单一职责：一个函数只做一件事；
- 可维护性：修改逻辑只需改一处；
- 可读性：代码意图清晰明了；
- 所有的问题都从头检查代码去深挖问题根源再解决，不要停留在表面修复；
- 不要通过模拟数据等任何方式掩饰问题的存在；
- 做任何事都优先检查已有的技能去处理；
- 不要自己造轮子，使用已有的库或框架；
- 优先使用Brainstorming跟我确认问题，不要直接开始编码；
- 可视化编辑中，都要考虑坐标转换问题；
- 不允许杜撰信息；
- SOLID 设计原则规则
1. 单一职责原则（SRP）
核心定义：单个软件单元（类 / 模块 / 函数）有且仅有一个变更原因，仅承担单一业务职责。强制约束：禁止单单元耦合多维度业务逻辑（如数据模型 + 持久化 + 通知逻辑合并）。校验规则：对该单元的修改，仅影响单一业务场景。
2. 开闭原则（OCP）
核心定义：软件实体对扩展开放、对修改关闭，新增功能通过扩展实现，不改动原有稳定代码。强制约束：禁止通过修改存量核心逻辑新增功能，优先通过抽象、多态实现能力扩展。校验规则：新增业务场景无需修改已上线的核心逻辑代码。
3. 里氏替换原则（LSP）
核心定义：子类可完全替换其父类，且不破坏原有程序的正确性与行为契约。强制约束：子类不得重写父类非抽象方法、不得抛出父类未声明的异常、不得违背父类输入输出约定。校验规则：所有父类出现的位置，替换为子类实例后程序无逻辑异常。
4. 接口隔离原则（ISP）
核心定义：客户端不应依赖其不需要的接口方法，禁止强制实现方实现无关接口。强制约束：拆分臃肿全量接口为最小粒度专用接口，实现类仅依赖自身需要的接口。校验规则：实现类无空实现、无抛出不支持异常的冗余接口方法。
5. 依赖倒置原则（DIP）
核心定义：高层模块不依赖低层模块，二者均依赖抽象；抽象不依赖细节，细节依赖抽象。强制约束：禁止高层模块直接实例化低层具体实现，优先通过依赖注入、面向接口编程解耦。校验规则：模块间顶层依赖为抽象接口，而非具体实现类。
